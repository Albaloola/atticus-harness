# Atticus Harness Live Run Hardening Report

Date: 2026-04-30
Scope: engineering report from the local live Napier/Anfal harness run, sanitized for GitHub. Provider secrets and raw private evidence text are intentionally omitted.

## Executive Summary

During the live DeepSeek/OpenRouter run, Atticus did not fail in one single dramatic way. It failed as layered control-plane brittleness: stale schema assumptions, gate deadlocks, provider credential errors being attributed to workers, retry loops that produced attention noise, malformed/over-broad worker outputs, unsafe citation semantics, and insufficient draft/redaction context. The latest code on `main` now fixes many of these concrete problems, but the live matter was stopped with `citation_audit` still missing and one repair task waiting for manual reducer review.

Most important conclusion: the harness now records the failures much better than it did, but the architecture still needs a serious pass to turn recorded blockers into deterministic repair workflows, especially for citation audit, source-material chunking, reducer review, and final quality gate progression.

## Bugs Faced And Fixed

### 1. Stale DB schema caused control-plane crashes

Observed failure: a resumed scheduler/orchestrator path attempted to write to `error_logs` and crashed with `sqlite3.OperationalError: no such table: error_logs`. This happened because the active matter DB had been created under an older schema while the code expected newer control-plane tables.

Fix status: fixed before this report in commit `8eed379` and reinforced in `700a9bb`. Write connections now self-apply additive schema and schema doctor verifies required tables/columns before stamping the schema version. This prevents the rescue path itself from depending on a table that may not exist.

Relevant areas: `atticus/db/repo.py`, `atticus/db/doctor.py`, schema tests.

### 2. Gates blocked progress without actionable repair

Observed failure: tasks reached `blocked` on missing certifications or incomplete dependencies and then simply sat there. The user had to ask the harness repeatedly to continue. This is a real loop hazard: gates are necessary, but a gate result must create an orchestrator-visible repair route.

Fix status: partially fixed. Gate blocks now emit `orchestrator.task_blocked`; worker failures/quarantines route to orchestrator events; loop guard escalation and maintenance events exist. The ledger now shows 115 `orchestrator.task_blocked` and 68 `orchestrator.worker_failed` events. Remaining design gap: some blocker repairs still require human/agent manual creation of a corrective task, especially citation-audit repair and stale coordinator task cleanup.

Relevant areas: `atticus/scheduler/gates.py`, `atticus/db/repo.py`, `atticus/agents/orchestrator.py`, `atticus/agents/maintenance.py`.

### 3. Provider credential/API failures were treated as worker failures

Observed failure: repeated OpenRouter HTTP 401 errors were recorded against many runnable tasks. This polluted task state and created noise that looked like worker failure. Exact grouped logs below show 33+35+10 occurrence batches for `OpenRouter provider call failed after dispatch: OpenRouter HTTP 401`, plus terminal preflight failures.

Fix status: fixed in commits `ad44c40` and `148a3e1`. The scheduler now performs preflight before leasing runnable tasks, classifies auth/credit errors as provider control-plane intervention, resolves provider attention after successful preflight, and avoids blaming individual workers for provider-key failure. Transient provider/malformed JSON failures are requeued instead of terminally blaming the worker.

Relevant areas: `atticus/scheduler/free_loop.py`, `atticus/db/repo.py`, `atticus/providers/live_readiness.py`, `tests/test_live_readiness.py`, `tests/test_free_loop.py`.

### 4. OpenRouter versioned actual model IDs were misread as failover

Observed failure: requested model `deepseek/deepseek-v4-flash` or `deepseek/deepseek-v4-pro` came back as `deepseek/deepseek-v4-flash-20260423` or `deepseek/deepseek-v4-pro-20260423` from providers such as DeepInfra/DeepSeek/AkashML/AtlasCloud. This is endpoint provenance, not an unsafe model downgrade.

Fix status: fixed in `atticus/providers/live_readiness.py` by allowing provider-versioned DeepSeek actual IDs to satisfy the requested Flash/Pro family. Tests cover the versioned endpoint behavior.

### 5. Result parser accepted unsafe or unusable artifact paths

Observed failure: workers emitted absolute local paths or otherwise unsafe `proposed_artifacts[0].path`. This is not only ugly: it risks writing outside the candidate artifact namespace and confuses provenance.

Fix status: fixed in `atticus/workers/result_parser.py`. Artifact paths now normalize known Atticus-local generated paths into `candidate/...` and reject arbitrary absolute paths, parent traversal, NUL bytes, home paths, and drive-like paths. Prompt instructions now require `candidate/<task_id>/...`.

Exact log examples include `worker output quarantined: proposed_artifacts[0].path must be a relative safe path`.

### 6. Derivative artifacts and rough drafts could leak into proof citations

Observed failure: workers cited extracted text artifacts, synthesis artifacts, rough drafts, or review artifacts as proof for legal/factual findings. A live citation audit specifically caught `art-a788d9fd8e8849dba5da5cf7a1ef238f` being used for key facts outside the work-order context. This weakens the evidence chain because OCR/extraction artifacts are reading aids, not independent proof.

Fix status: partially fixed and hardened in `148a3e1`. Citation target allow-lists now distinguish orientation-only artifacts from proof-allowed targets. Review/repair tasks get narrowly scoped permission to cite draft/review artifacts as the defect being repaired, not as primary evidence. Normal fact/procedure/risk findings require proof-allowed source/artifact/authority/claim/chronology evidence. Source-material prompts now tell workers to cite source IDs for OCR/extracted text facts.

Remaining serious fix: build deterministic quote/span support checking and source snapshot/chunk locators so the reducer can prove a quote exists in the cited source text, not merely that a source ID was cited.

### 7. Law findings could be supported without an authority citation

Observed failure: workers could describe legal duties using source/artifact citations instead of verified authority IDs. Citation audit found an Equality Act citation defect where the draft carried a malformed/truncated statutory reference.

Fix status: fixed in `atticus/workers/result_parser.py` and prompts. Supported `law` findings must cite at least one proof-allowed `authority` target. Prompts tell workers to use verified authority IDs for Equality Act 2010 sections 20/21 and section 91, and to mark unsupported legal propositions as uncertain rather than inventing authority.

### 8. Reducer certifications were too willing to certify defective audit outputs

Observed failure: audit/review tasks can produce a valid JSON packet that says “this fails,” and the reducer must not turn that into the certification that the audited thing passed. Citation audit, privacy audit, and final gate outputs need defect-aware certification withholding.

Fix status: fixed in `atticus/reducer/reducer.py`. `_certification_blocker()` now withholds `citation_audit`, `privacy_redaction_audit`, and `final_quality_gate` certifications when the packet contains defects/risk/redaction flags or failure content. Redaction review task types now map to `privacy_redaction_audit` only when clean.

### 9. Redaction workflow treated old unredacted material as a defect in the corrected target

Observed failure: a redaction verification task could compare against original unredacted material and incorrectly treat identifiers in the original as a defect unless the prompt and citation context were precise. It also found a real defect: an unsafe generic `[REDACTED]` placeholder where the safer contextual replacement was required.

Fix status: fixed/hardened. Prompts now say the original unredacted artifact is comparison evidence only; identifiers in the original are not defects unless the same unsafe identifier remains in the redacted target. Redaction fix/review task types can cite redacted draft and annotation artifacts as repair proof. The corrected redacted artifact was reduced and `privacy_redaction_audit` was issued at `2026-04-30T02:25:02+00:00`.

### 10. Context packs showed only artifact wrapper JSON, not the actual draft text

Observed failure: verifier/audit tasks could receive the first part of reducer wrapper JSON instead of the full proposed draft content. That makes a verifier blind to the actual text it is supposed to audit.

Fix status: fixed in `atticus/context/packs.py`. For full-text artifact types (`complaint_draft`, `draft`, `draft_complaint`, `redacted_draft`), artifact bundle context now extracts the actual proposed artifact content from reducer packet JSON and includes up to 16k characters with `content_chars`, `content_truncated`, and `content_source`.

### 11. Draft artifacts could be incomplete placeholder patches

Observed failure: a worker could submit a redacted draft or draft complaint with placeholder text such as “remaining unchanged” or “content omitted.” That is dangerous because downstream verification may certify a non-existent complete replacement.

Fix status: fixed in `atticus/workers/result_parser.py` and prompt adapters. Draft-like proposed artifacts are rejected if they contain placeholder patch language. Prompts instruct workers to submit complete replacement text or output a drafting note/follow-up task instead.

### 12. Auxiliary references could point to undefined citation IDs

Observed failure: a clean-looking redaction verification candidate listed an uncertainty referencing a citation id that did not exist. The parser accepted the packet, so the reducer had no reliable citation graph for auxiliary objects.

Fix status: fixed in `atticus/workers/result_parser.py`. `citation_ids` in `uncertainties`, `contradictions`, `risk_flags`, and `redaction_flags` must now refer to IDs present in the packet `citations`. Prompts were hardened accordingly. The bad live candidate was quarantined and rerun.

### 13. Worker-proposed follow-up tasks could be unscoped or impossible to run

Observed failure: workers proposed broad search/review tasks without source/artifact/task dependencies, and some imported tasks had `cost_limit_usd=0.0` even though the model policy estimated a nonzero cost. This produced avoidable blockers and capacity churn.

Fix status: fixed in `atticus/workers/proposed_tasks.py`. Scoped search/review task types must name concrete source/artifact/task dependencies. Imported task cost limits below provider estimated cost are normalized upward with an event trail (`proposed_task.cost_limit_normalized`, `task.cost_limit_normalized_after_import`).

### 14. Transient malformed JSON/provider responses caused hard worker failure

Observed failure: OpenRouter sometimes returned an incomplete or malformed JSON message, logged as `OpenRouter response did not contain a JSON message: Unterminated string...`. This was an external/transient provider response issue, not necessarily a task defect.

Fix status: fixed in `atticus/scheduler/free_loop.py`. Transient post-dispatch JSON/network/5xx errors now requeue; auth/credit failures remain terminal provider-control-plane blockers.

### 15. The harness still cannot fully finish without citation repair/reducer review

Observed remaining state: after the user asked to stop, the live task `napier-accommodation-arrears-repair-draft-citations-v1` was `reducer_pending`. It was routed to DeepSeek Pro and generated a candidate, but I did not manually reduce it because the user explicitly said stop/commit/push. The matter still lacks `citation_audit`; final quality gates remain blocked.
